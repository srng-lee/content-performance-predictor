## [2026-07-01 · Step2 /insight] 유사도 기준 설계
- 결정 내용:
  - type·channel 일치는 **하드 필터**(항상 요구, 완화 대상 아님) — 콘텐츠 형식 자체가 다르면 CTR 분포가 근본적으로 달라 비교 무의미
  - 하드 필터 통과 후보 내에서 topic_category(가중치 4) · has_emoji(가중치 2) · posting_hour(가중치 1) 일치 여부로 유사도 점수(0~7) 계산
  - 동점 시 headline_length 차이가 작은 쪽을 우선(제목 길이도 유사도의 일부로 간주)
  - 점수 내림차순 정렬 → 상위 3개를 유사 TOP3로 채택
- 선택한 이유:
  - 완화 순서를 posting_hour → has_emoji → topic_category로 정했으므로(가장 먼저 포기하는 조건이 posting_hour), 반대로 가중치는 topic_category를 가장 높게, posting_hour를 가장 낮게 주어 "먼저 포기하는 조건일수록 유사도 판단에서 비중이 작다"는 논리를 정확히 반영
  - 이진 가중치(4·2·1)를 쓰면 하나의 상위 조건 불일치가 하위 조건 전부 일치보다 항상 우선순위가 낮아지도록 보장되어, 실제 "조건을 하나씩 완화"하는 것과 수학적으로 동일한 정렬 결과를 만듦
  - 신규 5건 모두 type×channel 조합의 과거 후보가 6건 이상이라(예: 숏폼×유튜브 13건, 카드뉴스×인스타그램 22건) 하드 필터만으로도 TOP3 확보에 문제없음을 1단계 분석에서 확인
- 대안으로 고려했던 것:
  - type·channel까지 완화 대상에 포함(가이드 예시) → 미채택. 형식이 다른 콘텐츠(예: 블로그 vs 숏폼)를 비교하면 CTR 기준 자체가 달라 마케터에게 오히려 잘못된 벤치마크를 줄 위험 판단
  - 5개 조건 동일 가중치 → 미채택. "완화 순서"라는 설계 의도를 점수에 반영하지 못함

## [2026-07-01 · Step3 /generate] 개선 제안 선정 근거
- 결정 내용:
  - 신규 콘텐츠 1건마다 posting_hour·has_emoji·headline_length(블로그 한정) 3개 레버 각각의 "변경 시 CTR 개선폭(%p)"을 채널 단위 집계로 자동 계산
  - 개선폭(delta)이 큰 순으로 정렬해 상위 2개만 채택. 3개 레버 모두 개선 여지가 없으면(이미 최적 조건) type×channel 조합 비교로 대체 제안 1개를 보충
  - 모든 수치는 predict_pipeline.py의 집계 함수(compute_aggregates)에서 실시간 계산 — 특정 항목을 사람이 직접 골라 하드코딩하지 않음
- 선택한 이유:
  - "감이 아니라 데이터 근거"라는 채점 기준을 지키려면 제안 선택 자체도 규칙 기반이어야 함. delta 크기 순 정렬은 가장 단순하면서도 "가장 효과 큰 개선안부터 제시"라는 마케팅 의사결정 원칙과 일치
  - 채널 단위로 집계한 이유: company-info.md 예시("같은 유형·채널의 저녁 발행 평균 CTR")를 따르되, type×채널까지 쪼개면 일부 조합의 표본이 1~2건으로 줄어 평균의 신뢰도가 낮아지므로 채널 단위로 완화해 표본을 확보
- 대안으로 고려했던 것:
  - 5건 각각에 대해 사람이 임의로 제안 문구 작성 → 미채택. Standard 요구사항인 "새 CSV를 넣어도 동일 품질로 자동 생성"을 만족할 수 없음
  - type×channel 단위로 세분화한 시간대 평균 사용 → 미채택. 일부 조합 표본이 1건뿐이라 평균의 통계적 의미가 약함

## [2026-07-01 · Step5 /generate(Challenge)] 전략 기획안 우선순위 판단 기준
- 결정 내용: 우선순위 점수 = CTR 임팩트(45%) + 적용 범위(35%) + 실행 난이도(20%), 100점 만점으로 두 기획안을 채점해 P0/P1 부여
  - CTR 임팩트: 해당 전략이 근거로 삼는 실측 CTR 차이(%p)가 클수록 높은 점수
  - 적용 범위: 전체 정제 데이터(69건) 중 그 전략을 적용할 수 있는 콘텐츠 비율
  - 실행 난이도: 발행 시간·이모지처럼 즉시 적용 가능한 운영 규칙일수록 높은 점수, 콘텐츠 재제작이 필요하면 낮은 점수
- 선택한 이유:
  - CTR 임팩트에 가장 큰 비중(45%)을 둔 이유: 이 과제의 핵심 성과지표가 CTR이므로, 실제 클릭률 개선폭이 가장 직접적인 사업 가치
  - 적용 범위(35%)를 실행 난이도(20%)보다 높게 둔 이유: 아무리 쉬운 개선이라도 소수 콘텐츠에만 적용되면 전체 성과에 미치는 영향이 작음 — 반대로 넓게 적용되는 규칙은 실행이 다소 번거로워도 누적 효과가 큼
- 대안으로 고려했던 것:
  - 3개 지표 동일 가중치(33/33/34) → 미채택. CTR 개선이라는 과제 목적과 무관하게 점수가 산출될 위험

## [2026-07-01 · Step4 /generate] 리포트 섹션 순서 설계
- 결정 내용: 리포트 섹션 순서: 예상CTR범위(발행 여부 판단)→TOP3표(벤치마크 확인)→개선제안(실제 수정 작업) 순으로 배치. 마케터가 발행 전 의사결정할 때 거치는 순서와 동일하게 맞춤.
- 선택한 이유: 마케터가 신규 콘텐츠를 검토할 때 "발행할지 말지"부터 판단하고, 발행하기로 하면 "무엇과 비교되는지" 확인한 뒤, 마지막으로 "무엇을 고치면 더 나아지는지"를 본다는 의사결정 흐름과 리포트 읽는 순서를 일치시키기 위함
- 대안으로 고려했던 것: 개선 제안을 최상단에 배치 → 미채택. 아직 발행 여부도 정하지 않은 단계에서 개선안부터 보여주면 판단 순서가 뒤바뀜

## [2026-07-01 · /review 보완] 개선 제안 로직 예외 처리
- 결정 내용: `build_suggestions()`에서 신규 콘텐츠의 type×channel 조합이 이미 학습 데이터 내 최고 CTR 조합인 경우, "포맷 유지 권장" 메시지를 1번 제안으로 고정하고, 세부 레버(발행시간·이모지·제목길이) 중 실제 개선 여지가 있는 항목을 2번 제안으로 채택. 세부 레버도 전혀 없으면 같은 조합 내 topic_category별 engagement_rate 비교를 최종 폴백으로 사용
- 선택한 이유: 기존 로직은 "현재 조합보다 나은 조합"을 찾지 못하면 폴백이 작동하지 않아 개선 제안이 1개만 나오는 결함이 있었음(신규 콘텐츠 1 사례). "이미 최적 조합"이라는 사실 자체도 데이터 근거가 있는 유효한 제안이므로 이를 명시적으로 문구화해 모든 케이스에서 2개 제안이 채워지도록 수정
- 대안으로 고려했던 것: 세부 레버가 없을 때 억지로 임의 제안을 추가 → 미채택. 데이터 근거 없는 제안은 채점 기준("감 금지")에 위배되므로, engagement_rate라는 실제 지표로 대체
